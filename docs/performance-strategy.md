# Performance Testing Strategy

## Principle

Every profile answers a *different* question. Running "more load" is not a
strategy; matching a test shape to a business risk is.

| Profile | Question it answers | When | Gating? |
|---|---|---|---|
| **Smoke** | Does the system work at all, within SLO, at trivial load? | Every PR | ✅ Yes |
| **Load** | Do we meet SLOs at *expected* traffic? | Merge to main | ✅ Yes |
| **Stress** | Where is the saturation point — what breaks first? | Scheduled / manual | ❌ No |
| **Spike** | Can we survive and recover from a sudden burst? | Scheduled / manual | ❌ No |
| **Soak** | Do we degrade over time (leaks, creeping latency)? | Nightly / scheduled | ❌ No |
| **Degradation** | Can we *diagnose* a fault through the observability stack? | Demo / on-call drill | ❌ No |

## Why some profiles are non-gating

Stress, spike, and soak intentionally push the system past its comfort zone.
Their thresholds are **diagnostic**, not pass/fail — a stress test that "fails"
SLO has succeeded at its job (it found the limit). In CI these run with
`continue-on-error` so they publish reports without blocking merges. Treating an
exploratory test as a binary gate is a classic anti-pattern; we avoid it.

## Open vs closed models

- Smoke uses a **closed model** (`constant-vus`): a fixed number of virtual
  users, each looping. Good for a quick "is it alive" check.
- Load/stress/spike use an **open model** (`ramping-arrival-rate`): we control
  *arrival rate* (requests/s) independent of how fast the system responds. This
  is the realistic model for user traffic — real users don't slow their arrival
  because your server is busy — and it lets a slow system build a backlog
  instead of artificially throttling the offered load.
- Soak/degradation use `constant-arrival-rate` to hold a steady offered rate
  over a long window.

## Journey weighting

The smoke/load journey is the full self-management flow (login → invoices →
plan change → payment) so a single VU iteration exercises every tier. Per-request
`journey:<name>` tags let us evaluate each journey's SLO independently from one
combined run, rather than maintaining four separate scripts.

## Test data

50 synthetic customers, deterministic credentials (`user_NNN` / `password123`),
generated by [`infra/postgres/generate-seed.mjs`](../infra/postgres/generate-seed.mjs).
The k6 user picker mirrors the seed's plan assignment so plan-change targets are
always valid (avoids self-inflicted 4xx noise that would corrupt the error SLI).

## Reports & evidence

Run with `--summary-export` to capture machine-readable results:

```
k6 run --summary-export=tests/k6/reports/smoke-summary.json tests/k6/scenarios/smoke.js
```

These JSON summaries are the CI artifacts and the raw material for the
interview walkthrough.
